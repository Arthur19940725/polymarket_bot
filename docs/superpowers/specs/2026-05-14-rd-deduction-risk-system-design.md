# 研发费用加计扣除风险管理体系搭建方案（设计文档）

| 项 | 内容 |
|---|---|
| 文档版本 | v1.0 |
| 设计日期 | 2026-05-14 |
| 适用对象 | 单一企业自用 / 大型软件 · 互联网 · IT 企业 / 上市公司或拟上市企业 |
| 适用政策 | 财税〔2015〕119 号 + 国家税务总局公告 2017 年第 40 号 + 国家税务总局公告 2023 年第 7 号 |
| 加计比例 | 一般软件企业 100% 加计扣除（2023 年起常态化） |
| 体系周期 | 24 个月分三阶段落地（M0–M6 闭环建设 / M7–M12 内控成熟 / M13–M24 数据驱动） |

---

## 0. 设计背景与目标

### 0.1 用户画像
- 单一大型企业（>500 人）
- 软件 / 互联网 / IT 行业，研发投入 >1 亿
- IT 系统基础薄弱（无 ERP 深度集成、工时仍依赖周报 / Excel）
- 同时面对税务稽查、年报审计、IPO / 持续披露 三重压力

### 0.2 优先解决的三大风险点
| # | 风险 | 体系定位 |
|---|------|----------|
| ② | 研发 vs 非研发活动边界不清 | 由**判定层**解决 |
| ③ | 人员工时分摊证据链薄弱 | 由**证据层**解决 |
| ⑧ | 组织职责不清（财务一人扛） | 由**治理层**解决 |

### 0.3 体系总体目标
1. **合规目标**：经得起税务稽查 + 年报审计 + IPO 披露三重检验
2. **效率目标**：月度归集 5 个工作日内闭环、年度汇算清缴前 60 天完成自查
3. **风险目标**：历史年度无重大追溯调整、未来年度红灯指标 <10%

### 0.4 总体架构（"治理 + 轻量数据中台"双轮驱动）
```
┌─────────────────────────────────────────────────┐
│  ① 治理层  委员会 + 办公室 + RACI 矩阵          │
├─────────────────────────────────────────────────┤
│  ② 判定层  判定矩阵 + 6 项负面清单 + 三道闸门    │
├─────────────────────────────────────────────────┤
│  ③ 证据层  四张主表 + 客观证据强制 + 低代码平台  │
├─────────────────────────────────────────────────┤
│  ④ 内控层  三级审核 + 月/季/年三阶检查           │
├─────────────────────────────────────────────────┤
│  ⑤ 演进层  24 个月三阶段路线图                   │
├─────────────────────────────────────────────────┤
│  ⑥ KPI 层  合规 / 效率 / 风险三类 12 项指标      │
├─────────────────────────────────────────────────┤
│  ⑦ 预案层  税务稽查 / 历史追溯 / 政策变动 / BCP   │
└─────────────────────────────────────────────────┘
```

## 1. 治理层（组织架构与职责分工）

### 1.1 三层组织架构

**决策层 · 研发费用加计扣除管理委员会**（季度例会）
- 组长：CFO / 财务副总裁
- 委员：研发 VP、HR 总监、内审负责人、法务总监
- 职责：审定政策、裁决重大争议、批准项目认定、年度复盘

**执行层 · 加计扣除办公室**（虚拟编制，常设）
- 主任：财务总监指定 1 人（建议税务经理 / 财务经理兼任）
- 成员：财务专员 1、研发 PMO 接口人 1、HR 接口人 1
- 职责：日常运营、月度归集、政策跟踪、培训、稽查应对

**执行节点**（分散各部门）
- 研发部门：项目经理（PM）对应项目
- HR：薪酬专员对接工资数据
- 财务：成本会计对接费用归集
- 法务：审核委外 / 合作研发合同
- 内审：季度抽查 + 年度专项

### 1.2 RACI 矩阵（关键 7 项任务）

| 任务 | 研发 PM | HR | 财务 | 法务 | 内审 | 加计办 | 委员会 |
|------|---------|----|----|------|------|--------|--------|
| 研发项目立项判定 | R | C | C | I | I | **A** | I |
| 研发人员工时填报 | **R** | C | I | — | I | A | — |
| 工时月度复核 | C | R | C | — | I | **A** | I |
| 工资 / 费用月度归集 | I | C | **R** | — | C | A | — |
| 委外 / 合作研发合同审核 | C | I | C | **R** | I | A | I |
| 年度汇算清缴申报 | C | C | **R** | C | C | A | I |
| 重大争议裁决 | C | C | C | C | C | C | **R/A** |

R=负责执行 / A=最终问责 / C=协助咨询 / I=知情

### 1.3 三个常态化机制
1. **月度归集对账会**（加计办 + 研发 PMO + HR + 财务，月初 5 日内）
2. **季度风险评审会**（加计办向委员会汇报，含红黄绿灯仪表盘）
3. **年度专项自查 + 复盘**（内审主导，汇算清缴前 60 天启动）

### 1.4 治理层配套文件（4 份）
- 《研发费用加计扣除管理委员会章程》
- 《加计扣除办公室运作细则》
- 《研发费用加计扣除管理办法》
- 《加计扣除工作 RACI 责任矩阵》

## 2. 判定层（研发活动认定标准 + 项目闸门）

### 2.1 政策依据
- 财税〔2015〕119 号 + 国税总局公告 2017 年第 40 号（核心定义 + 负面清单）
- 国税总局公告 2023 年第 7 号（最新申报口径）
- 与软件企业认定（财税〔2012〕27 号）、高新企业认定（国科发火〔2016〕32 号）三套口径**并行管理**

### 2.2 研发活动判定矩阵（软件企业专用）

| 类别 | 是否可加计 | 判定标准（满足 ≥2 条） | 典型场景 |
|------|------------|--------------------------|----------|
| **A. 真研发** | ✅ 全额计入 | ① 技术不确定性 ② 新算法/架构/协议 ③ 立项书 + 验收标准 ④ 新功能/新性能/新产品 | 新产品 0→1、核心算法升级、架构重构 |
| **B. 准研发** | ⚠️ **默认不计入，委员会逐项裁决** | 需留存技术不确定性说明 | 重大版本升级、性能优化、跨平台移植 |
| **C. 技术服务 / 客户定制** | ❌ | 客户合同已约定交付物，无技术不确定性 | 实施部署、二开、定制功能 |
| **D. 运维 / 日常迭代** | ❌ | Bug 修复 / 已知技术常规应用 | Bug fix、UI 微调、SaaS 运维 |

> **关键设计**：B 类默认不计入——宁可少算，不可错算。

### 2.3 负面清单（119 号文，强制排除）
1. 产品（服务）的常规性升级
2. 对某项科研成果的直接应用
3. 商品化后的售后技术支持
4. 现有产品工艺 / 材料的简单改变
5. 市场调查 / 效率调查 / 管理研究
6. 工业（服务、流程）的常规设计

### 2.4 项目三道闸门

| 闸门 | 时点 | 必交产出物 | 评审方 |
|------|------|------------|--------|
| **G1 立项** | 项目启动前 | 立项申请书 + **技术不确定性说明** + 预算 / 工时估算 + 核心人员名单 | 研发主管 + 加计办 + 财务，争议升委员会 |
| **G2 中检** | 每半年 / 关键节点 | 进展报告 + 工时偏差 + 范围变更说明 | 加计办 + 研发 PM |
| **G3 结项** | 完成后 30 天内 | 结项报告 + 验收证明 + 产出物清单 + 实际工时 / 费用汇总 | 研发主管 + 加计办 + 财务 |

> "技术不确定性说明"是稽查现场被问"为什么这算研发"的第一抗辩文件。

### 2.5 边界案例库
- 第一年录入 ≥30 例
- 每例字段：项目编码 / 业务描述 / 争议点 / 委员会裁决 / 政策依据 / 留存证据清单
- 后续新项目可直接比对

### 2.6 判定层配套文件（5 份）
- 《研发活动判定矩阵与操作指引》
- 《研发项目立项申请书模板》（含技术不确定性说明）
- 《研发项目中检报告模板》
- 《研发项目结项报告模板》
- 《研发活动边界案例库》（动态）

## 3. 证据层（四张主表 + 低代码闭环）

### 3.1 数据模型

```
        ┌─────────────────┐
        │ ① 项目主数据表  │ ← 主键：项目编码（如 RD-2026-001）
        └─────────────────┘
                │
        ┌───────┼───────┐
        ▼       ▼       ▼
   ② 人员-项目映射  ③ 工时表  ④ 产出物表
```

#### 表① 项目主数据
| 字段 | 必填 | 备注 |
|------|------|------|
| 项目编码 | ✅ | 全公司唯一 |
| 项目名称 / 类别 (A/B) | ✅ | 类别由委员会裁定 |
| 立项 / 结项日期 | ✅ | |
| 项目负责人工号 | ✅ | |
| 立项申请书 PDF | ✅ | 附件存档 |
| 技术不确定性说明 | ✅ | 核心抗辩文件 |
| 关联 Jira Project Key | ✅ | **核心打通字段** |
| 关联成本中心 | ✅ | **核心打通字段** |

#### 表② 人员-项目映射（按月维护）
- 员工工号 + 项目编码 + 月份
- 分配工时占比（月初 PM 填、月末复核）
- 该员工当月项目数 **>3 触发预警**

#### 表③ 工时填报（日 / 周填报）
| 字段 | 必填 | 客观证据 |
|------|------|----------|
| 员工工号 / 日期 / 项目编码 / 工时（≤8h/日） | ✅ | |
| 工作内容描述（≥20 字） | ✅ | |
| **关联 Jira Ticket ID** | ⭐ | **三选一强制** |
| **关联 Git Commit Hash** | ⭐ | 同上 |
| **关联会议纪要 ID** | ⭐ | 同上 |
| 主管复核时间戳 | ✅ | 周五前完成 |

> **核心机关**：⭐三选一缺失的工时项，月度复核时**自动剔除**（前 3 个月过渡期只预警、第 4 个月起强制）。

#### 表④ 产出物登记
代码仓库 / 技术文档 / 软著 / 发明专利 / 测试报告 / 论文 / 行业标准

### 3.2 系统集成（轻量化，**不改造源系统**）

```
[HR 系统] ──CSV 月度导入──┐
                          ▼
[Jira/禅道] ──API 拉取─→ [飞书多维表格 / 简道云]
                          ▲          │
[GitLab/GitHub] ──Webhook─┘          │
                                     ▼
                          [财务 ERP：成本中心归集]
```

**集成方式**：HR → CSV 月度导入；Jira → API 每日拉取；Git → Webhook 推送；财务 → CSV 月度导出。

### 3.3 月度归集 5 步流程

| 日期 | 步骤 | 责任方 |
|------|------|--------|
| T+1 | 数据归集（HR/Jira/Git 导入低代码） | 加计办 |
| T+2 | 工时合规校验（剔除无证据 / 单日 >8h / 项目数 >3） | 系统自动 |
| T+3 | PM 月度复核 + 修正 | 各项目 PM |
| T+4 | 工资分摊计算（工时 × 工资基数） | 加计办 + HR |
| T+5 | 财务归集到成本中心 + 加计办汇总 | 财务 + 加计办 |

### 3.4 低代码平台选型
- **首选**：飞书多维表格 + 应用引擎（年费 8–15 万）
- 备选：钉钉宜搭 / 简道云 / 明道云
- 决策原则：公司主用什么生态，就选什么

### 3.5 证据链交叉验证规则（系统自动）

| 规则 | 触发 | 处置 |
|------|------|------|
| 工时填了但当日无 Git/Jira/会议痕迹 | 🟡 | PM 复核时补充说明或修正 |
| 单人单月工时 <100h 但分摊 100% | 🔴 | 强制下调比例 |
| Jira 类型为 Bug/Support 但计入研发 | 🔴 | 自动剔除 |
| 项目当月无任何产出 | 🔴 | 委员会复审项目状态 |
| 人员当月项目数 >3 | 🟡 | PM 出具兼岗说明 |

### 3.6 证据层配套文件（6 份）
- 《研发项目数据模型说明书》
- 《低代码平台搭建手册》
- 《工时填报操作指南》（员工 + PM）
- 《月度归集 SOP》
- 《数据接口对接说明》（HR/Jira/Git）
- 《证据链交叉验证规则说明》

## 4. 内控层（三级审核 + 月/季/年三阶检查）

### 4.1 三级审核流程

| 层级 | 责任人 | 频率 | 内容 | 留痕 |
|------|--------|------|------|------|
| **一级 自审** | 员工填、PM 签 | 日填 / 周签 | 工时真实性 + 证据完整性 | 系统时间戳 |
| **二级 复核** | 加计办 | 每月 5 日前 | 数据合规性（红黄灯）+ 跨项目人员异常 | 月度归集报告 |
| **三级 终审** | 财务总监（季）+ 委员会（半年/年） | 季 / 年 | 总量异常 + 政策合规 + 留存备查完备 | 评审纪要 |

### 4.2 月度对账 5 项必对（任一不平衡，月度归集报告不得签发）

| # | 对账项 | A 源 | B 源 | 容差 |
|---|--------|------|------|------|
| 1 | 研发人员名单 | HR 在职库 | 表② | 0% |
| 2 | 工时总量 | 表③ | 工作日 × 人数 × 8h | ±10% |
| 3 | 工资归集 | 工时 × 工资基数 | HR 工资总账 | 0% |
| 4 | 项目编码 | Jira 列表 | 表① | 100% 覆盖 |
| 5 | 成本中心 | ERP 研发科目 | 表①归集 | ±1% |

### 4.3 季度风险评审仪表盘（7 项指标 × 红黄绿灯）

| 指标 | 绿灯 | 黄灯 | 红灯 |
|------|------|------|------|
| 研发费用 / 营收 vs 预算偏差 | ±5% | ±5–15% | >±15% |
| 工资薪金占比 | 60–75% | 50–60% 或 75–85% | <50% 或 >85% |
| **其他相关费用占比** | <4% | 4–5% | ≥5%（法定上限）|
| **工时无客观证据占比** | <5% | 5–15% | >15% |
| 人员当月项目数 >3 比例 | <10% | 10–20% | >20% |
| 立项 / 结项报告齐备率 | 100% | 90–100% | <90% |
| B 类项目数 / 总项目数 | <20% | 20–40% | >40% |

> **阈值前 3 个月作为基线测算期、半年后委员会复审调整。**

### 4.4 年度专项自查（12 步，T-60 天起）

```
T-60: 内审启动，加计办移交全年数据
T-55: 抽样审计（按金额 / 风险分层，覆盖 ≥30%）
T-45: 留存备查 10 类齐备性核对
T-35: 高新 / 软件企业认定口径交叉比对
T-25: 测算加计扣除金额 + 敏感性分析
T-15: 自查报告初稿，委员会复审
T-10: 整改 / 补正
T-5:  财务定稿
T:    汇算清缴申报
T+30: 留存备查打包入库（10 年保管）
T+60: 复盘会
T+90: 下年度政策跟踪 + 制度修订
```

### 4.5 留存备查 10 类（按 40 号公告 + 内部补充）
1. 立项书 / 2. 研发人员名单 / 3. 委外 · 合作合同 / 4. 外聘人员 + 仪器设备 · 无形资产费用分配说明 / 5. 集中研发分摊表 / 6. "研发支出"辅助账及汇总表 / 7. 鉴定意见（若有）/ 8. 省局规定其他资料 / 9. 工时台账 + 客观证据导出包（内部）/ 10. 年度自查报告 + 委员会评审纪要（内部）

> **保管期：10 年**（自当年汇算清缴期结束之日起）

### 4.6 三色风险预警机制
- 🔴 红灯 → 加计办 24h 内出《风险事件报告》→ 委员会专题会议 → 限期整改
- 🟡 黄灯 → 纳入下月监控清单 + 月度报告专项说明
- 🟢 绿灯 → 季度评审常规通报

### 4.7 内控层配套文件（5 份）
- 《研发费用归集三级审核流程图与操作细则》
- 《月度对账 SOP（5 项必对）》
- 《季度风险评审仪表盘模板 + 指标定义》
- 《年度专项自查工作手册（12 步清单）》
- 《留存备查档案管理办法（10 类 / 10 年）》

## 5. 演进层（24 个月落地路线图）

### 5.1 总体路线

```
M0 ────── M6 ────── M12 ────── M18 ────── M24
│ Phase 1 │ Phase 2  │      Phase 3       │
│ 闭环建设 │ 内控成熟  │     数据驱动        │
```

### 5.2 Phase 1：闭环建设（M0–M6）

| 时点 | 里程碑 | 交付物 |
|------|--------|--------|
| M0 | 体系立项 + CFO 签发委员会章程 | 章程 + project charter |
| M1 | 治理层落地（委员会、加计办、RACI 培训） | 4 份治理文件 + 培训记录 |
| M2 | 判定层落地（管理办法 + 判定矩阵 + 负面清单宣贯） | 5 份判定文件 + 全员考试 |
| M2–M3 | 证据层搭建（飞书四张主表 + 接口对接） | 低代码平台 + SOP |
| M3 | 工时填报上线（**过渡期**：只预警不剔除）+ 首次季度评审（看基线） | 首月工时数据 + 仪表盘 v1 |
| M4 | **过渡期结束**，工时强制剔除规则启用 + 历史项目补录 | 红黄灯生效公告 |
| M5 | 内控层试运行（首次月度对账 5 项 + 二级复核） | 月度归集报告 |
| M6 | **首次半年报**：Phase 1 复盘 + Phase 2 启动 | 半年评估报告 |

**Phase 1 DoD**：
- ✅ 全部研发人员入系统填报
- ✅ 项目主数据 100% 录入

| S4 核心人员流动 | 加计办负责人 / 关键 PM 离职 | 🟡 |

### 7.2 S1 · 税务稽查应对预案

**黄金 24 小时清单**
```
T+0:    收到通知 → 1h 内上报 CFO + 委员会
T+2h:   成立专项应对小组（加计办 + 财务 + 法务 + 外部税务师）
T+4h:   评估稽查范围 / 涉及年度 / 重点项目
T+8h:   调取留存备查档案（项目库 + 月度归集报告 + 自查报告）
T+24h:  出具《应对策略备忘》报委员会
```

**三道防线**
1. 留存备查 10 类齐备 = 70% 胜算
2. 项目立项书 + 技术不确定性说明（B 类核心抗辩）
3. 边界案例库 + 委员会裁决记录（证明体系性）

**关键纪律**
- ❌ 不得现场临时编造 / 补签文件
- ❌ 不得未经加计办 + 法务同意单独答复
- ✅ 所有书面回复必经委员会 + 法务复核
- ✅ 涉及金额 >500 万 / 罚款风险时强制外聘税务师事务所

### 7.3 S2 · 历史年度追溯预案

```
发现 → 加计办 5 日内出《风险事项报告》
     → 委员会评估三条处置路径：
        ① 主动更正申报（金额小、瑕疵明确）
        ② 等待税务机关检查（金额不大、瑕疵模糊）
        ③ 聘请外部税务师 + 法务出具专业意见（金额大、定性争议）
     → 同步排查类似瑕疵的扩散面
     → 整改纳入下季仪表盘红灯项跟踪
```

### 7.4 S3 · 政策变动响应
