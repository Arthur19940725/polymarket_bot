## 第一章 总则

**第 1 条 目的**
明确飞书多维表格与 HR 系统、Jira、GitLab/GitHub 三类源系统的数据集成方式、字段映射、安全要求、运维与异常处置，确保数据闭环可追溯、可审计。

**第 2 条 核心原则**
1. **轻接入**：不改造任何源系统，仅做"读取 + 汇总"
2. **只读权限**：所有接口仅授予只读凭证；读取范围最小化
3. **数据脱敏**：源系统中的身份证、手机号等无关字段不进入飞书
4. **变更可审计**：所有接口调用 / Webhook 回调有日志，留存 ≥6 个月
5. **失败可降级**：接口故障时人工 CSV 导入应作为兜底方案

## 第二章 接口架构总览

```
   [HR 系统]──── CSV 月度导入 ──────┐
                                    ▼
   [Jira/禅道]── API 每日拉取 ──→ [飞书多维表格]
                                    ▲           │
   [GitLab/GitHub]── Webhook ──────┘           │ T+5 月度导出
                                                ▼
                                       [财务 ERP：成本中心归集]
```

| 接口 | 方向 | 触发方式 | 频率 |
|------|------|----------|------|
| HR → 飞书 | 入 | 人工 CSV 导入 | 每月 1 日 |
| Jira → 飞书 | 入 | 飞书自动化定时拉取 | 每日 02:00 |
| GitLab/GitHub → 飞书 | 入 | Webhook 实时推送 | 实时（commit 时触发）|
| 飞书 → 财务 ERP | 出 | 人工 CSV 导出 | 每月 T+5 |

## 第三章 HR 月度 CSV 接口

### 3.1 数据范围

| 字段 | 必填 | 用途 |
|------|------|------|
| 工号 | ✅ | 主键 |
| 姓名 | ✅ | 显示 |
| 部门 | ✅ | 组织归属 |
| 岗位 | ✅ | 角色判定 |
| 在职状态 | ✅ | 在职 / 离职 / 调岗 |
| 入职日期 | ✅ | 在职月数 |
| 月度工资基数 | ✅ | 工资分摊计算（仅工资薪金，**不含**奖金 / 期权 / 非货币性补贴）|
| 邮箱 | ✅ | Git author email 匹配 |
| 直接主管工号 | | 复核流程 |

### 3.2 脱敏要求

源系统导出时**必须排除**：
- ❌ 身份证号
- ❌ 手机号
- ❌ 家庭住址
- ❌ 银行卡号
- ❌ 紧急联系人

### 3.3 文件格式

- 编码：**UTF-8 (BOM)**
- 分隔符：英文逗号
- 文件名：`HR_YYYYMM.csv`（如 `HR_202611.csv`）
- 首行表头：字段中文名

**示例**：
```csv
工号,姓名,部门,岗位,在职状态,入职日期,月度工资基数,邮箱,直接主管工号
E001234,王某某,推荐算法部,算法工程师,在职,2024-03-01,35000,wangxx@example.com,E000001
E001235,李某某,推荐算法部,后端工程师,在职,2025-06-15,28000,lixx@example.com,E000001
```

### 3.4 操作流程

每月 1 日 09:00 前：
1. HR 接口人在 HR 系统执行预设报表"研发加计扣除月度导出"
2. Excel 中人工脱敏（删除敏感列）
3. 另存为 UTF-8 CSV
4. 上传到飞书云空间 `/07-数据归档/YYYY-MM/HR_YYYYMM.csv`
5. 加计办主任接收通知，在飞书多维表格执行批量导入

### 3.5 校验规则

导入时校验：
- ☐ 工号唯一性
- ☐ 邮箱格式合法
- ☐ 月度工资基数 > 0
- ☐ 在职状态在 `{在职, 离职, 调岗, 试用}` 范围内
- ☐ 导入前后员工总数差异 ≤ 5%（异常波动告警）

异常处置：违反任一规则的记录单独标黄，由 HR 接口人核实修正后重新导入。

## 第四章 Jira API 接口

### 4.1 API 类型

使用 Jira Cloud REST API v3 或 Jira Server REST API v2（按公司部署版本）。

### 4.2 凭证管理

- IT 在 Jira Admin 创建 Service Account："飞书集成-加计扣除"
- 生成 API Token（只读权限，仅 Browse Projects + View Issues）
- Token 保存在飞书集成平台的 **加密变量**中，不出现在任何代码 / 日志 / 文档明文
- Token 每 90 天轮换一次

### 4.3 接口规格

**请求**：
```
GET https://【Jira 域名】/rest/api/3/search

Headers:
  Authorization: Bearer <API_TOKEN>
  Content-Type: application/json

Query:
  jql=updated>=-1d ORDER BY updated DESC
  fields=summary,status,issuetype,assignee,reporter,project,created,updated
  maxResults=100
  startAt=0
```

**响应字段映射**：

| Jira 字段 | 飞书表⑤字段 | 处理 |
|-----------|--------------|------|
| `key` | Ticket ID | 主键 |
| `fields.project.key` | Jira Project Key | 直接 |
| `fields.summary` | Summary | 直接 |
| `fields.status.name` | Status | 中英映射（Open→打开等）|
| `fields.issuetype.name` | Type | 同上 |
| `fields.assignee.emailAddress` | Assignee 工号 | 通过邮箱→工号映射 |
| `fields.reporter.emailAddress` | Reporter 工号 | 同上 |
| `fields.created` | 创建时间 | ISO 8601 |
| `fields.updated` | 最后更新时间 | ISO 8601 |

### 4.4 自动化工作流

在飞书自动化助手中配置：

```
触发：每日 02:00 定时
步骤 1：HTTP GET 调用 Jira API
       └── 增量参数：jql=updated>=-1d
步骤 2：循环分页（startAt += 100）直到取完
步骤 3：逐条调用"添加/更新记录"动作到飞书表⑤
       └── 按 Ticket ID 去重 / 更新
步骤 4：记录同步日志到「集成日志」表
步骤 5：失败时发飞书消息到加计办群
```

### 4.5 邮箱→工号映射

由于 Jira assignee 字段为邮箱，需要在飞书自动化中维护映射：
- 数据源：HR 月度 CSV 中的"邮箱"+"工号"字段
- 飞书表「员工映射」自动维护
- 工作流中查询此表完成转换

### 4.6 失败重试

- 单次同步失败：等待 30s 后重试，最多 3 次
- 连续 3 天同步失败：发飞书消息升级至加计办主任 + IT 接口人
- 失败期间数据缺口：恢复后通过手动触发"全量补同步"工作流

### 4.7 性能考虑

- 全公司 Jira ticket 数预估 50000+，每日新增/更新约 500–2000 条
- 增量同步预计耗时 < 5 分钟
- 月度全量同步（建议每月 1 日做一次）预计 < 2 小时

## 第五章 GitLab / GitHub Webhook 接口

### 5.1 Webhook 类型

**事件**：Push、Merge Request（推荐两者都开启）

### 5.2 配置步骤

在每个 GitLab/GitHub Project 设置中：

1. 进入 Settings → Webhooks
2. 添加新 Webhook：
   - **URL**：飞书自动化助手生成的接收端点
   - **Secret Token**：与飞书侧约定的鉴权字符串（每个项目可独立）
   - **Trigger events**：Push events、Merge request events
   - **SSL verification**：启用
3. 点击「Test」验证连通性

### 5.3 Webhook Payload 字段映射

**GitLab Push Webhook 示例**：

```json
{
  "object_kind": "push",
  "project": {
    "name": "recommend-multimodal",
    "namespace": "ai-team"
  },
  "ref": "refs/heads/feature/clip-finetune",
  "commits": [
    {
      "id": "a1b2c3d4e5f6...",
      "message": "feat(clip): add ViT-B/16 backbone for e-commerce fine-tune",
      "author": {
        "email": "wangxx@example.com"
      },
      "timestamp": "2026-09-15T14:23:45+08:00",
      "added": ["..."],
      "modified": ["..."],
      "removed": ["..."]
    }
  ]
}
```

**字段映射**：

| Webhook 字段 | 飞书表⑥字段 | 处理 |
|--------------|--------------|------|
| `commits[].id` | Commit Hash | 主键 |
| `project.name` | 仓库名 | 直接 |
| `ref` | 分支 | 截取 `refs/heads/` 后部分 |
| `commits[].message` | 提交信息 | 直接 |
| `commits[].author.email` | Author Email | 直接 |
| 通过邮箱查映射 | Author 工号 | 调用员工映射表 |
| `commits[].timestamp` | 提交时间 | ISO 8601 |
| `len(added) + len(modified)` | 涉及文件数 | 计算 |
| 需调用 Git API 补查 | 净增加行数 | 异步补查（可选）|

### 5.4 鉴权与安全

- Secret Token 验证：飞书侧收到 webhook 后通过 HMAC-SHA256 校验签名
- 签名头：GitLab `X-Gitlab-Token` 或 GitHub `X-Hub-Signature-256`
- 校验失败：返回 401，记录到失败日志
- HTTPS 强制：仅接受 https 推送

### 5.5 失败处置

- GitLab/GitHub 内置重试机制：失败时 1 分钟 / 5 分钟 / 30 分钟三次重试
- 飞书侧失败：返回 5xx 触发 GitLab 重试
- 长期失败：在飞书"集成日志"表中标记，加计办与 IT 联合排查
- 数据缺口恢复：手动调用 Git API 拉取指定时段 commit 补录

### 5.6 数据量预估

- 公司全部仓库 commit 量：约 500–2000 次/日
- Webhook 实时推送，无延迟
- 飞书表⑥年增量约 50 万条，需配套清理策略：保留近 3 年明细 + 历史月度汇总

## 第六章 财务 ERP 出口（飞书 → ERP）

### 6.1 输出内容

每月 T+5 月度归集报告签发后，加计办主任执行：

1. 从飞书多维表格导出《人员工资分摊明细表》（CSV）
2. 从飞书多维表格导出《项目费用归集汇总表》（CSV）
3. 文件上传至飞书云空间 `/07-数据归档/YYYY-MM/`

### 6.2 财务侧操作

- 财务成本会计接收 CSV，对照 ERP 研发科目核对
- 在 ERP 中按项目编码（成本中心代码）完成成本归集
- 在 ERP「研发支出辅助账」登记
- 月底执行 ERP 与飞书的成本中心余额对账（容差 ±1%）

## 第七章 接口监控与运维

### 7.1 监控指标

| 指标 | 告警阈值 | 通知对象 |
|------|----------|----------|
| Jira 同步连续失败 | ≥3 天 | 加计办主任 + IT |
| Jira 同步耗时 | >30 分钟 | IT |
| Git Webhook 单日推送量 | < 历史均值 50% | 加计办 + IT |
| HR CSV 未按时导入 | > T+1 12:00 | CFO（升级）|
| 飞书表存储用量 | >80% 配额 | IT |

### 7.2 日志保留

- 全部接口调用 / Webhook 回调 / CSV 导入记录到「集成日志」表
- 字段：时间 / 接口名 / 状态 / 数据条数 / 耗时 / 错误信息
- 保留期：≥6 个月（线上）+ 3 年（归档）

### 7.3 季度巡检

加计办每季度组织一次集成健康度巡检：
- 接口可用率 ≥ 99.5%
- 数据完整性核查（与源系统抽样对比）
- 性能基线复核
- 安全凭证轮换检查

## 第八章 安全与合规

### 8.1 凭证管理

- 所有 API Token / Webhook Secret 由 IT 安全管理
- 飞书集成平台启用加密变量功能
- 凭证轮换周期 ≤ 90 天
- 离职 / 调岗的 IT 接口人凭证立即吊销

### 8.2 数据保护

- 工资基数等敏感字段在飞书设置字段级权限（仅加计办 + 财务可见）
- 飞书账号必须启用 SSO + 2FA
- 数据导出操作需加计办主任审批

### 8.3 审计

- 内审季度抽查接口运维日志
- 留存备查档案包含全年接口健康度报告

## 第九章 应急与降级

### 9.1 接口故障应急

| 故障 | 降级方案 |
|------|----------|
| Jira API 长期失败 | 切换为人工 CSV 导出（每周 1 次）|
| Git Webhook 不可用 | 切换为 IT 定时脚本拉取 |
| 飞书自动化故障 | 切换为人工执行 |
| HR CSV 导入失败 | 推迟月度归集 1 日，CFO 备案 |

### 9.2 月度归集延期

- 接口故障导致月度归集超过 T+5 → 加计办主任向 CFO 汇报，可申请延期至 T+7
- 延期超过 T+10 → 提交委员会专题会议
