# 工时填报操作指南（员工版）

> **致研发部全体同事**：
> 本指南是每天 3 分钟必读手册。**填好工时 = 公司每年保住 2500 万节税 = 你的项目继续有钱**。
> 看完一次，养成习惯，终生受益。

---

## 一、你需要知道的最重要 5 件事

1. **每天下班前 5 分钟，在飞书「研发加计扣除运行底表」填好今天的工时。** 别拖到周末，凑不出来。
2. **每条工时记录必须关联以下至少一项客观证据**（⭐ 三选一）：
   - Jira Ticket ID（最常用）
   - Git Commit Hash
   - 会议纪要 ID（飞书会议自动生成）
3. **过渡期：M3–M5 三个月内（即 2026-08 至 2026-10）**，缺证据只会黄灯提示、不会剔除；
   **2026-11-01 起进入强制期**，没有客观证据的工时项**月底自动从加计扣除基数里剔除**。
4. **工作内容描述写清楚，至少 20 字**。不要写"日常工作"、"开发"、"开会"这种空话。
5. **PM 每周五前会复核签字**。退回的工时要在下周一前改完，否则计入当月异常名单。

## 二、填报操作（手机 / 电脑均可）

### 2.1 入口

**手机**：飞书 App → 工作台 → 「研发加计扣除」→ 工时填报
**电脑**：浏览器打开 飞书.work / 工作台 → 「研发加计扣除」→ 工时填报

### 2.2 填报步骤（3 分钟内完成）

#### 第 1 步：选项目

- 下拉选择今天工作的项目（系统只显示**你这个月被 PM 分到的项目**）
- **看不到目标项目？** → 找 PM 月初把你加到「人员-项目映射」里

#### 第 2 步：填工时

- 单日总工时 ≤ 8 小时
- 同一项目可只填一条；多个项目分多条填
- 加班单独填，需附加班审批单（找主管走加班流程）

#### 第 3 步：写工作内容描述

**好示例 ✅**：
> 今天完成多模态推荐召回模块的 CLIP 双塔模型训练脚本，对比了 ResNet-50 和 ViT-B/16 两种视觉编码器在电商数据集上的召回率，ViT 高 3 个百分点，决定后续以 ViT 为主。

**坏示例 ❌**：
- "开发"（2 字，太短）
- "写代码"（3 字，看不出做了什么）
- "日常工作"（4 字，空话）

#### 第 4 步：⭐ 关联客观证据（必填至少一项）

- **Jira Ticket ID**：你今天做的工作对应的 Jira ticket 编号（格式如 `RECOMMEND-1234`）
- **Git Commit Hash**：你今天提交的 commit hash（前 7 位以上即可，如 `a1b2c3d`）
- **会议纪要 ID**：飞书会议结束后自动生成的纪要 ID

> 三个不需要都填——**任意一个就行**。
> 如果都没有怎么办？请重新审视：你今天的"研发工作"是否真的存在客观痕迹？没有的话，可能这部分时间不属于研发活动。

#### 第 5 步：保存

点保存。系统会立刻显示这条记录的状态：
- 🟢 通过校验，等待 PM 复核
- 🟡 黄灯：缺客观证据（过渡期可暂存，强制期会拒绝保存）
- 🔴 红灯：工时超 8h / 项目不在你的映射里 / 描述太短

## 三、常见疑问

### Q1：我同时做多个项目，怎么填？

A：分多条填。例如今天 4h 做 A 项目、4h 做 B 项目：填两条记录，每条 4h，各自关联自己的证据。

### Q2：今天开会、没写代码，怎么关联证据？

A：填飞书会议的纪要 ID。开会前用飞书会议起会，会议结束后系统会自动出纪要。即使纯会议也算研发活动（设计评审 / POC 讨论 / 算法 review 都算）。

### Q3：我今天做 bug 修复，能不能填到研发项目？

A：**不能。** Bug 修复属于 D 类运维，不算研发。bug 修复时间填到所在部门的"日常运维"任务上（不进入加计扣除基数）。

### Q4：客户驻场实施 / 客户问题排查的时间呢？

A：**不能填到研发项目。** 属于 C 类技术服务。这些时间走客户项目工时（项目管理系统）。

### Q5：我加班到深夜，工时怎么填？

A：单日总工时上限 8h。超出部分需走加班审批流程，并在备注里说明"加班 ___ h，审批单号 ___"。**长期加班的工时合规性也会被加计办关注**，过高加班会触发红灯。

### Q6：忘了昨天填，今天补？

A：可以补填**最近 7 天内**的工时。超过 7 天的系统不允许补填——这是为了保证证据时效性。

### Q7：周末 / 节假日做研发，要填吗？

A：要填。研发活动不分工作日 / 周末，只看时间是否真的投入。但要走加班流程。

### Q8：我换项目了，原项目还没结项，还能填吗？

A：找 PM 把你从原项目的"人员-项目映射"中移除，新月份的映射加到新项目。**已经填好的历史工时不受影响**。

### Q9：填错了怎么办？

A：当天可自行修改。如果已经过了 PM 复核签字，要找 PM 退回，或加计办在月底前修正。**月度归集签发后不可改**——所以日常勤填、勤改。

### Q10：我对"什么算研发"还是不清楚？

A：找你的 PM 看立项申请书的"技术不确定性说明"。如果你在做的工作和说明书里描述的一致，就属于研发；不一致或边界模糊，问 PM 或加计办（飞书群「加计办公开答疑」）。

## 四、客观证据的本质

很多同事问："为什么必须关联 Jira / Git / 会议？"

**答案**：因为周报和主管签字**在税务稽查现场没有任何抗辩力**。稽查员会问：
- "这个员工 8 月份说自己做了 160 小时研发，证据呢？"
- "周报是事后写的，可以伪造。"
- "主管签字是主观判断，没有时间戳。"

而 Jira 状态变更、Git commit 时间戳、飞书会议纪要 ID 都是**系统自动产生、不可伪造、有秒级时间戳**的客观痕迹——这才是真证据。

你每天关联一下 Jira/Git/会议，就是在帮公司守住每年 2500 万的节税。

## 五、奖惩与连带责任

### 5.1 红灯指标会被关注

- 月度红灯次数 > 3 → 飞书消息提示你修正
- 月度红灯次数 > 5 → 加计办联系你的主管
- 持续 2 个月红灯 → 视为不配合，部门绩效连带扣分

### 5.2 严禁伪造工时

- 编造 / 代填 / 倒签工时 = 违反《研发费用加计扣除管理办法》第 30 条
- 一经发现：① 工时全部剔除 ② 通报批评 + 部门负责人约谈 ③ 严重者按公司奖惩管理办法处理
- 涉嫌违法（如骗取国家税收优惠）→ 移交司法机关

### 5.3 配合好就有正向反馈

- 工时填报合规率高的部门 → 季度评审"绿星"部门
- 配合最积极的 PM → 年度优秀 PM 提名加分
