# 工时填报操作指南（PM 版）

> **致研发项目经理（PM）**：
> 你是加计扣除体系在一线落地的"守门员"。研发项目的真实性、人员投入的合理性、客观证据的完整性，全靠你的复核把关。
> 这份指南告诉你三件事：**月初要做什么、每周要做什么、月底要做什么。**

## 二、月初任务（每月 1–3 日完成）

### 2.1 维护「人员-项目映射」表

**操作**：飞书多维表格 → 表② → 你负责的项目

每个项目你需要确认：
- ☐ 本月有哪些员工参与
- ☐ 每个员工本月在你的项目上的**分配工时占比**（按预估）
- ☐ 离开 / 新加入项目的员工调整

### 2.2 占比怎么填

| 员工情况 | 建议占比 |
|----------|---------|
| 全职在你的项目上 | 100% |
| 一半时间在你项目、一半在另一项目 | 50%（另一项目的 PM 也填 50%）|
| 偶尔咨询 / 评审，每周 ≤ 4h | 10% 或不入映射 |
| 部门主管，只挂名不真做 | **不要填入**，否则触发红灯 |

> **关键规则**：同一员工当月所有项目占比之和应 ≤ 100%。系统会自动校验。

### 2.3 兼岗预警

- 任一员工当月项目数 > 3 → 系统自动 🟡 黄灯
- 你需要在该员工的映射记录"备注"字段出具《兼岗合理性说明》（≥50 字）
- 例："王某某为算法骨干，本月同时支援推荐 / 搜索 / 风控三大项目的算法 review，每个项目占比 30%、20%、20%，剩余 30% 为部门通用工作。"

### 2.4 截止时间

- **每月 3 日 23:59 前**完成映射维护
- 超期未维护会触发飞书消息提醒；连续 2 个月超期，加计办联系研发主管

## 三、每周任务（每周五 17:00 前完成）

### 3.1 复核本周工时

**操作**：飞书多维表格 → 表③ →「我的项目工时」视图 → 筛选「主管复核状态 = 待复核」

#### 复核 5 步法

**第 1 步：看时长**
- 单日 > 8h？要么有加班单，要么是错填
- 全月低于 100h 但占比 100%？很可能是占比错填，需调整

**第 2 步：看描述**
- < 20 字？退回，要求补充
- 写"日常工作 / 开发 / 测试"等空话？退回
- 看起来像 bug 修复 / 客户支持？退回（不属于研发）

**第 3 步：看客观证据**
- Jira/Git/会议三个字段**至少要有一项**
- 过渡期（M3–M5）：缺证据时给员工**温和提示**，鼓励养成习惯；可以暂时通过
- 强制期（M6 起）：缺证据系统自动拒绝，员工已无法保存，到不了你这里

**第 4 步：看活动类型是否属于研发**
- 客户定制需求？→ C 类，退回，让员工填到客户项目
- 已知技术常规应用？→ D 类，退回
- 不确定？→ 与加计办商量

**第 5 步：签字**
- 通过：把"主管复核状态"改为"通过"，系统自动记录时间戳
- 退回：改为"退回"，并在备注里写明原因；员工会收到飞书提醒

### 3.2 退回不是冒犯

退回工时记录是 PM 的本职，不是给员工"小鞋穿"。
- 你今天宽松通过一条不合规工时，相当于让公司今年多承担**真金白银的稽查风险**
- 员工被退回也不是处罚，是帮 ta 养成正确习惯
- 一线员工不熟悉政策口径，靠 PM 把关

## 四、月底任务（T-3 ~ T 日）

### 4.1 配合加计办月度对账

加计办在月初 5 日内启动月度归集，期间可能找你确认：
- 工时合规校验中的红黄灯项是否能修正
- 某员工占比与实际严重不符的修正
- 客观证据缺失项的最后补救（强制期前还有补救余地）

**响应时限**：加计办询问后 **1 个工作日内**回复

### 4.2 出具偏差说明（必要时）

- 本月项目预算执行 vs 实际偏差 > 20% → 加计办会找你出说明
- 重大范围变更 → 走《研发项目中检报告》流程

### 4.3 异常处置 SLA

| 异常类型 | 处置时限 |
|----------|----------|
| 红灯工时项（必须剔除）| T+3 日前 PM 确认 |
| 黄灯工时项（建议修正）| T+5 日前给意见 |
| 兼岗超 3 个项目 | T+3 日前出具说明 |
| 员工反馈复核退回有异议 | T+5 日前与加计办协商 |

## 六、常见疑难与处置

### Q1：员工说"今天就是开了一天会，没有 Jira/Git"

A：让员工填飞书会议的纪要 ID。
- 飞书会议结束后会自动出纪要，纪要里有 meeting ID
- 即使是临时讨论会、纯口头沟通，也应开飞书会议记录
- 如果团队还没养成"开会必开飞书会议"的习惯，**这是 PM 要立刻改变的**

### Q2：员工说"我在做调研，看论文，没有 Git commit"

A：调研属于研发活动的合规部分，但必须有客观痕迹。建议：
- 把调研笔记 / 论文综述写到 Confluence，**作为 Git commit 提交**（如`docs/research/multimodal-survey.md`）
- 或者作为一个 Jira ticket 跟踪（"调研 - 多模态推荐 SOTA 论文综述"）

### Q3：员工兼岗超 3 个，怎么写说明？

A：诚实说明，不要硬凑。常见合理场景：
- "该员工为架构师，本月支援 X / Y / Z 三大项目的架构评审，每周 1–2 次"
- "该员工本月正在交接，从 X 项目逐步切换到 Y 项目"

如果是不合理的（员工只是名义挂名），那本来就不应该填入"映射"，应当移除。

### Q4：员工历史工时填得乱，怎么补救？

A：分情况：
- 过渡期内（M3–M5）：可以补正，把客观证据补上
- 强制期（M6 起）：不可补，月底自动剔除

### Q5：和员工对"算不算研发"有争议？

A：找加计办商量。重大争议升委员会。
- 不要 PM 自己拍板"算"——风险全在你身上
- 也不要简单粗暴"不算"——影响员工士气

### Q6：项目快结项了，工时还能补吗？

A：结项后 30 天内必须完成结项报告（按《结项报告模板》）。报告里的"实际工时"是最终口径，结项后不可修改。所以结项前一定要让员工把工时补齐。

## 八、给 PM 的工作流模板

### 月初任务清单（每月 1 日发起）

- [ ] 1 日：查看本月项目状态、人员变动
- [ ] 2 日：与团队成员 1on1 确认本月预期工时占比
- [ ] 3 日：在飞书多维表格完成「人员-项目映射」录入
- [ ] 3 日：兼岗超 3 个的员工写《兼岗合理性说明》

### 每周复核清单（每周五发起）

- [ ] 周四晚：飞书提醒团队成员"明天前补齐工时"
- [ ] 周五 14:00：进入飞书多维表格「我的项目工时」视图
- [ ] 周五 14:00–17:00：按 5 步法复核每条记录
- [ ] 周五 17:00：完成"主管复核状态"批量签字

### 月底协作清单（T-3 ~ T 日）

- [ ] T-3：加计办发出月度对账请求，1 日内回复
- [ ] T-2：完成本项目所有异常工时的处置
- [ ] T-1：项目里程碑节点检查（必要时启动中检报告）
- [ ] T：项目状态更新到表①
