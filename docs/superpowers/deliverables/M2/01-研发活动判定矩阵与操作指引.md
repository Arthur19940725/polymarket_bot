## 第一章 总则

**第 1 条 目的**
为统一全公司研发活动认定口径，解决"研发 vs 非研发"边界长期争议（spec 风险点 ②），制定本指引。本指引为《研发费用加计扣除管理办法》第三章的具体操作细则。

**第 2 条 适用范围**
本指引适用于公司本部及纳入合并报表范围的下属研发主体的所有研发活动认定，包括但不限于：
1. 项目立项（G1）阶段的类别判定
2. 月度归集时的活动归属判定
3. 边界争议事项的提交与裁决

**第 3 条 总体原则**
1. **实质重于形式**：以研发活动的客观属性判定，而非以部门名称或人员编制判定
2. **保守优先**：边界模糊时，默认不计入加计扣除，由委员会逐项裁决
3. **客观证据优先**：立项书、技术不确定性说明、Git 产出物等客观痕迹优先于主观周报
4. **一致性原则**：相同性质活动应作相同认定，沉淀至边界案例库

## 第二章 判定矩阵（四象限）

### 第 4 条 四象限模型

```
                        技术不确定性强
                              │
                  B 类         │         A 类
              （准研发-默认排除）│       （真研发-全额计入）
                              │
   非合同约束 ─────────────────┼───────────────── 合同约束
                              │
                  D 类         │         C 类
            （运维-不计入）    │   （技术服务/定制-不计入）
                              │
                        技术不确定性弱
```

### 第 5 条 四类详细判定标准

#### A. 真研发（全额计入加计扣除）

**判定标准**（至少满足 2 条 + 必须满足第 1 条）：
1. ⭐ **存在技术不确定性**——立项时无法确定能否完成、或如何完成
2. 涉及新算法、新架构、新协议、新核心机制
3. 有正式立项书 + 验收标准 + 技术不确定性说明
4. 预期产出新功能、新性能、新产品（不是 UI 改版或 bug 修复）
5. 技术方案需要试错、迭代、对比验证

**软件企业典型场景**（8 例）：
| # | 场景 | 判定理由 |
|---|------|---------|
| 1 | 基于 LLM 自研 RAG 框架并实现 query 重写算法 | 新算法 + 技术不确定性（向量检索效果是否优于关键词）|
| 2 | 新一代 API 网关：从单体改为基于 eBPF 的高性能转发层 | 新架构 + 性能不确定 + 需 POC 验证 |
| 3 | 推荐系统从协同过滤升级为多模态深度学习召回 | 新算法 + 召回率不确定 + 离线 A/B 实验 |
| 4 | 自研分布式时序数据库（应对客户数据量级超过 ClickHouse 上限）| 新核心机制 + 写入性能不确定 |
| 5 | 跨平台移动端 SDK：基于 React Native 自研性能优化框架 | 跨平台技术不确定性 + 自研框架 |
| 6 | 自研容器编排系统（K8s 不满足公司特定 GPU 调度需求）| 新协议 + 调度算法不确定 |
| 7 | 基于隐私计算的联邦学习平台 | 新协议（同态加密 + 安全多方计算）+ 收敛性不确定 |
| 8 | 自研代码生成 IDE 插件（AI Copilot 类）| 新算法（context window 优化）+ 准确率不确定 |

#### B. 准研发（默认不计入，委员会逐项裁决）

**判定标准**（满足以下任一项需委员会裁决）：
1. 重大版本升级（v2.x → v3.x，涉及架构调整）
2. 性能优化达到指标级别（如吞吐量提升 ≥30%、延迟降低 ≥50%）
3. 跨平台移植（如从 PC 端到移动端、从 x86 到 ARM）
4. 重大重构（如单体拆微服务、关系型数据库迁移到分布式）
5. 对外开源贡献并形成行业标准

**关键设计**：
> **默认不计入。** 若 PM 认为应计入，须在立项书附《技术不确定性说明》（≥500 字），由委员会专题裁决。

**软件企业典型场景**（5 例）：
| # | 场景 | 委员会通常裁决 |
|---|------|---------------|
| 1 | 核心数据库引擎从 MySQL 迁移到 TiDB | ⚠️ 取决于是否解决新问题（如海量分布式事务）|
| 2 | 单体后端拆分微服务（200+ 服务）| ⚠️ 取决于是否涉及自研服务治理框架 |
| 3 | 前端框架从 Vue 2 升级到 Vue 3 | ❌ 一般为常规升级 |
| 4 | 性能优化使页面加载从 3s 降至 1s | ⚠️ 取决于优化是否包含新算法 |
| 5 | PC 端产品移植到 iOS / Android | ⚠️ 取决于是否解决跨平台新问题 |

#### C. 技术服务 / 客户定制（不计入加计扣除）

**判定标准**：
1. 按客户合同需求定制开发
2. 合同已明确约定交付物与验收标准
3. 不涉及技术不确定性（已知技术的客户化应用）
4. 收入计入主营业务收入

**软件企业典型场景**（6 例）：
| # | 场景 | 判定理由 |
|---|------|---------|
| 1 | 客户 X 项目按合同交付的客制化报表模块 | 合同约束 + 已知技术 |
| 2 | 给客户 Y 做的 SSO 单点登录对接 | 已有协议（OAuth2 / SAML）应用 |
| 3 | 给客户 Z 做的数据导入工具（Excel → 数据库）| 通用 ETL 应用 |
| 4 | 客户驻场实施 + 配置 + 调优 | 实施服务，非研发 |
| 5 | 客户专属功能开发（仅 1 个客户使用）| 定制 + 无通用性 |
| 6 | 给客户做的培训材料 + 用户手册 | 文档服务，非研发 |

> **特殊情况**：若客户定制项目中**确实包含了自研新算法 / 新架构、且后续可推广到通用产品**，可向委员会申请将这部分**研发性工作时长**单独剥离计入加计扣除——但需提供客观证据（如剥离后的代码仓库 / 技术文档 / 后续产品化记录）。

#### D. 运维 / 日常迭代（不计入加计扣除）

**判定标准**（明确排除）：
1. Bug 修复（无论严重程度）
2. 已知技术的常规应用
3. UI 微调 / 文案修改 / 配置变更
4. 监控告警 / 日志分析 / 性能调优（非算法级）
5. 数据库 DBA 日常维护
6. SaaS 平台日常运维

**软件企业典型场景**（8 例）：
| # | 场景 | 判定理由 |
|---|------|---------|
| 1 | 登录页 UI 改版 + 5 个 P3 bug 修复 | UI + bug fix |
| 2 | 把按钮颜色从蓝改成绿 | 文案 / 样式调整 |
| 3 | 数据库索引优化 | 已知技术应用 |
| 4 | 监控告警阈值调整 | 运维 |
| 5 | 升级第三方库到新版本（无 breaking change）| 常规升级 |
| 6 | SaaS 平台日常补丁发布 | 运维 |
| 7 | 客户问题排查与回复 | 售后支持 |
| 8 | API 文档勘误 | 文档维护 |

## 第三章 6 项负面清单（强制排除）

### 第 6 条 来自财税〔2015〕119 号文

| # | 负面清单 | 软件企业典型表现 | 处置 |
|---|----------|-----------------|------|
| 1 | 企业产品（服务）的常规性升级 | 版本迭代、UI 改版、小功能新增 | 一律 D 类，不得加计 |
| 2 | 对某项科研成果的直接应用 | 已有算法 / 开源库的常规调用 | 一律 D 类 |
| 3 | 企业在商品化后为顾客提供的技术支持活动 | 客户问题排查、实施支持、配置变更 | 一律 C 类，不得加计 |
| 4 | 对现存产品、服务、技术、材料或工艺流程进行的重复或简单改变 | 配置调整、参数优化 | 一律 D 类 |
| 5 | 市场调查研究、效率调查或管理研究 | 用户调研、A/B 测试运营分析、流量分析 | 一律 D 类 |
| 6 | 作为工业（服务）流程环节或常规的质量控制、测试分析、维修维护 | 常规 QA 测试、回归测试 | 一律 D 类（但研发性测试除外，如新算法准确率测试可计入）|

### 第 7 条 执行规则

1. 负面清单具有**绝对排除效力**——即使 PM 认为属于研发，也不得加计。
2. 立项申请书填写时，PM 必须勾选"不属于 6 项负面清单"声明，否则立项不予通过。
3. 月度归集时发现违反负面清单的工时项，加计办有权直接剔除。

## 第四章 操作流程

### 第 8 条 立项判定流程

```
研发主管发起立项
        ↓
PM 填写立项申请书 + 技术不确定性说明
        ↓
PM 自评类别 (A/B/C/D)
        ↓
加计办初审（10 个工作日内）
        ↓
   ┌──── A 类 ────→ 研发主管 + 加计办 + 财务三方签字通过
   │
   ├──── B 类 ────→ 委员会专题会议裁决
   │
   ├──── C 类 ────→ 不进入加计扣除项目库，按客户项目管理
   │
   └──── D 类 ────→ 不立项，作为日常运维处理
```

### 第 9 条 月度归集判定流程

```
研发员工日填工时（关联项目编码）
        ↓
PM 周末复核签字
        ↓
加计办月初校验：
  ① 项目类别是否为 A 或委员会通过的 B
  ② 工时是否关联客观证据（Jira/Git/会议）
  ③ 是否触犯负面清单
        ↓
异常项标记红黄灯并处置
        ↓
合格项纳入工资分摊计算
```

### 第 10 条 争议升级路径

| 争议级别 | 触发场景 | 处理路径 | 时限 |
|----------|----------|----------|------|
| 一级 | PM 与加计办对类别判断不一致 | 加计办主任协调 | 5 个工作日 |
| 二级 | 加计办与研发主管 / 财务总监意见相左 | 提交委员会专题会议 | 14 个工作日 |
| 三级 | 委员会内部意见分歧无法解决 | CFO 行使一票否决 / 邀请外部税务师出具意见 | 30 个工作日 |

## 第五章 边界案例库管理

### 第 11 条 案例库定位

边界案例库（spec 风险点 ② 核心抗辩资产）是判定矩阵的**动态延伸**，记录所有 B 类委员会裁决案例及典型争议案例，作为：
1. 后续相似项目的判定参照
2. 税务稽查现场的抗辩依据
3. 体系自我学习与迭代的素材

### 第 12 条 案例库录入

每个 B 类裁决案例须录入飞书多维表格「边界案例库」表，字段包括：
1. 案例编号（`BC-YYYY-NNN`）
2. 项目编码
3. 业务描述（≥200 字）
4. 争议点
5. 委员会裁决日期
6. 裁决结论（A/B-通过 / B-否决 / C / D）
7. 政策依据条款
8. 留存证据清单（关联立项书、技术不确定性说明 PDF 路径）
9. 备注

### 第 13 条 案例库扩充目标

| 时点 | 目标累积 |
|------|----------|
| M3 末 | 5 例 |
| M6 末 | 15 例 |
| M9 末 | 30 例 |
| 每年 | 新增 ≥20 例 |

## 第六章 与高新认定 / 软件企业认定的口径并行

### 第 14 条 三套口径差异

| 口径 | 政策依据 | 研发费用定义范围 | 加计扣除适用 |
|------|----------|-------------------|--------------|
| 加计扣除 | 财税〔2015〕119 号 + 国税总局公告 2017 年第 40 号 | 6 类（人员人工 / 直接投入 / 折旧 / 无形资产摊销 / 试验费 / 其他相关 5% 限额）| ✅ |
| 高新认定 | 国科发火〔2016〕32 号 | 8 类（含管理费用 20% + 其他费用 20%）| ❌ |
| 软件企业认定 | 财税〔2012〕27 号 | 类似高新但有差异 | ❌ |

### 第 15 条 口径并行管理

1. 加计办按"**加计扣除口径**"为主，对外申报。
2. 高新认定 / 软件企业认定时另行计算并出具《三套口径对照表》，避免相互打架。
3. **横向比对预警**：若某项目在加计扣除中归为 A 类、但在高新中归为非研发，必须出具差异说明。
