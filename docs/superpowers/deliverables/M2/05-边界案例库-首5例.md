## 一、案例库使用说明

### 1.1 定位

- 判定矩阵的**动态延伸**——把每次委员会裁决沉淀为可复用的判例
- 后续相似项目的**判定参照**——新项目可先查案例库再决定是否升委员会
- 税务稽查现场的**抗辩资产**——证明公司有系统化、非临时拼凑的判定体系

### 1.2 字段定义

| 字段 | 必填 | 说明 |
|------|------|------|
| 案例编号 | ✅ | 格式：`BC-YYYY-NNN`（BC = Boundary Case）|
| 项目编码 | ✅ | 关联具体项目 `RD-YYYY-NNN` |
| 业务描述 | ✅ | ≥200 字客观描述 |
| 争议点 | ✅ | PM 申请 vs 委员会观点 |
| 委员会裁决日期 | ✅ | YYYY-MM-DD |
| 裁决结论 | ✅ | A 通过 / B 通过 / B 否决 / C / D |
| 政策依据条款 | ✅ | 具体到文件 + 条款 |
| 留存证据清单 | ✅ | 立项书 / 技术说明 / 会议纪要 PDF 路径 |
| 后续类似项目处理建议 | ✅ | 给后来者的指引 |

### 1.3 检索方式

1. 按业务类型（推荐 / 搜索 / 风控 / 平台 / 移动端 ...）
2. 按裁决结论（A 通过 / B 通过 / B 否决 ...）
3. 按政策依据条款
4. 按关键词

### 案例 BC-2026-001：核心数据库从 MySQL 迁移到 TiDB

| 字段 | 内容 |
|------|------|
| 项目编码 | RD-2026-007 |
| 业务描述 | 公司主交易系统数据库从单机 MySQL 8.0 迁移到分布式 TiDB 6.5。迁移背景：MySQL 单机峰值 QPS 已达 5 万、写入瓶颈明显、订单量 Q3 预期翻倍。迁移内容：① 数据库选型与对比测试 ② 业务表分片策略设计 ③ 分布式事务方案（基于 TiDB 内置 + 应用层补偿）④ 业务代码适配（SQL 语法差异修正）⑤ 数据迁移与一致性校验工具开发 ⑥ 灰度切流与回滚预案。预计周期 6 个月，核心团队 8 人。 |
| 争议点 | PM 主张 A 类（分布式事务自研补偿、迁移工具自研）；加计办初审认为属 B 类（核心是已有产品的应用迁移，TiDB 是开源成熟方案）。|
| 委员会裁决日期 | 2026-09-15 |
| 裁决结论 | **B 类通过**——但仅允许将"分布式事务补偿框架"+"数据一致性校验工具"两个子模块（占总工时 35%）计入加计扣除。其余 65% 工时（业务代码适配 / 灰度切流 / 配置调优）作为 D 类不计入。 |
| 政策依据条款 | 财税〔2015〕119 号文（研发活动定义）+ 国税总局公告 2017 年第 40 号文第六条第二项（"对现存产品的简单改变"不属于研发）|
| 裁决理由 | 1. TiDB 本身为开源成熟产品，迁移本身属于常规升级 2. 但"分布式事务补偿框架"涉及业务特定的对账逻辑，存在技术不确定性，可计入 3. "数据一致性校验工具"涉及大规模数据 diff 算法，可计入 4. 其余子模块属于已知方案的工程实施 |
| 留存证据 | RD-2026-007 立项书 PDF / 技术不确定性说明 / 委员会裁决纪要（2026-09-15）/ 工时拆分明细 |
| 后续建议 | 类似"成熟开源产品迁移"项目，默认 D 类，仅允许"自研补丁 / 自研工具"部分作为 B 类申请。|

### 案例 BC-2026-002：自研多模态推荐召回系统

| 字段 | 内容 |
|------|------|
| 项目编码 | RD-2026-012 |
| 业务描述 | 公司主推荐系统从协同过滤升级为多模态（行为 + 图像 + 文本）深度学习召回。涉及自研 CLIP-style 双塔模型电商域微调、自研 Faiss-HNSW 索引结构、自研多模态融合 ranking 模块。立项目标：召回率 65%→85%、CTR 3.2%→4.0%。预计周期 6 个月。|
| 争议点 | PM 主张 A 类；初审无异议。委员会复核确认。|
| 委员会裁决日期 | 2026-10-10 |
| 裁决结论 | **A 类通过**——全额计入加计扣除。|
| 政策依据 | 财税〔2015〕119 号文（研发活动定义，符合"创造性运用科学技术新知识"）|
| 裁决理由 | 1. 多模态融合方案在电商垂直场景无成熟先例（业界 SOTA 通用领域有效但电商不确定）2. 自研索引结构需在 100ms 延迟约束下达 1 亿向量召回，存在工程不确定性 3. POC 阶段方案对比明确显示技术不确定性 4. 立项书技术不确定性说明充分（820 字）|
| 留存证据 | RD-2026-012 立项书 / POC 对比报告 / 算法实验记录 / Git 仓库快照 |
| 后续建议 | "自研算法 + 工程不确定性"双满足的项目为典型 A 类，可参照本案例直接通过初审。|

### 案例 BC-2026-003：客户 X 项目定制报表模块

| 字段 | 内容 |
|------|------|
| 项目编码 | （不分配 RD 编码，按客户项目管理）|
| 业务描述 | 与客户 X 签订软件实施合同，合同金额 200 万。其中包含为客户定制开发的 15 张专属报表（用户行为分析 / 销售趋势 / 库存周转等），周期 3 个月，团队 4 人。报表完全按客户提供的 PRD 开发，技术方案为标准 BI 引擎 + SQL 模板。|
| 争议点 | PM 试图申报"客户场景算复杂，应作为研发"；加计办否决。|
| 委员会裁决日期 | 2026-10-20 |
| 裁决结论 | **C 类否决**——不进入加计扣除项目库。|
| 政策依据 | 财税〔2015〕119 号文负面清单第 6 条（工业（服务）流程环节或常规的质量控制、测试分析、维修维护、调查研究）+ 第 40 号公告（客户定制开发不属于研发活动）|
| 裁决理由 | 1. 合同已明确约定交付物与验收标准 2. 报表逻辑虽复杂但全部基于已有 BI 引擎 + SQL 模板 3. 无任何技术不确定性 4. 收入计入主营业务收入（实施服务）|
| 留存证据 | 客户合同 / 客户 PRD / 报表交付物清单 |
| 后续建议 | "客户合同约束 + 已知技术应用"组合一律 C 类。即使客户场景看起来"复杂"，只要技术方案已知，就不属于研发。|

### 案例 BC-2026-004：前端框架升级 Vue 2 → Vue 3

| 字段 | 内容 |
|------|------|
| 项目编码 | （不分配 RD 编码）|
| 业务描述 | 公司主站前端从 Vue 2 升级到 Vue 3，涉及 200+ 组件改造、Composition API 迁移、状态管理库切换、构建工具升级（Webpack → Vite）。周期 4 个月，团队 6 人。|
| 争议点 | 研发部主张"框架升级涉及大量代码改造，应作为 B 类研发"；加计办否决。|
| 委员会裁决日期 | 2026-11-05 |
| 裁决结论 | **D 类否决**——属于常规升级，不立项进入研发项目库。|
| 政策依据 | 财税〔2015〕119 号文负面清单第 1 条（产品的常规性升级）|
| 裁决理由 | 1. Vue 3 为成熟开源框架，升级路径业界有标准答案 2. 工作量大不等于技术不确定性 3. 无新算法、新协议、新机制产生 4. 类似性质：将 Web 应用从 Bootstrap 3 升级到 Bootstrap 5——同样不算研发 |
| 留存证据 | 研发部立项申请 / 加计办否决说明 / 委员会裁决纪要 |
| 后续建议 | "成熟框架版本升级"一律 D 类。判断口径：是否需要"试错 + 备选方案"？如果答案是"按官方迁移指南做即可"，那就不是研发。|

### 案例 BC-2026-005：自研容器编排（K8s GPU 调度增强）

| 字段 | 内容 |
|------|------|
| 项目编码 | RD-2026-019 |
| 业务描述 | 公司 AI 训练平台需要管理 200+ 张 A100/H100 GPU，但 K8s 原生 GPU 调度器（device-plugin）无法满足：① GPU 拓扑感知（NVLink 优先同卡组）② 多任务时分复用（MIG 切分动态调整）③ 抢占式调度（高优先级训练任务抢占低优先级）。自研 K8s 调度器扩展（scheduler plugin），并贡献部分代码到 Kubernetes 社区。周期 8 个月，团队 6 人。|
| 争议点 | PM 主张 A 类（自研调度算法）；加计办初审认为可能是 B 类（基于 K8s scheduler 框架扩展，不是从零自研）。|
| 委员会裁决日期 | 2026-12-15 |
| 裁决结论 | **A 类通过**——全额计入加计扣除。|
| 政策依据 | 财税〔2015〕119 号文（研发活动定义）|
| 裁决理由 | 1. 拓扑感知 + MIG 动态切分 + 抢占式调度，业界没有现成解决方案 2. 调度算法本身存在显著技术不确定性（需要对比多种打分函数、做大量仿真）3. 对 K8s 社区有贡献，技术新颖性获得外部认可 4. 立项书技术不确定性说明详尽（含 4 套算法对比方案）|
| 留存证据 | RD-2026-019 立项书 / 调度算法对比仿真报告 / 社区 PR 链接 / 性能测试报告 |
| 后续建议 | "在开源框架上做核心算法扩展"且"算法本身有技术不确定性"——A 类通过。区别于 BC-2026-001（应用迁移属 B/D 类）。|

## 四、维护规则

### 4.1 录入触发

1. 所有 B 类委员会裁决案例 → **必须录入**
2. A 类争议较大的（初审不通过、复议通过） → **建议录入**
3. C / D 类典型否决（容易混淆边界） → **建议录入**
4. 历史年度被税务稽查问询的项目 → **必须录入**

### 4.2 录入责任人

- 加计办主任：审核与最终录入
- 各项目 PM：提供项目描述与争议点初稿
- 委员会秘书：会议纪要中的裁决理由摘录

### 4.3 复审与更新

1. 每季度由加计办主任带领回看一遍案例库，识别是否有政策变化导致案例需要调整
2. 每年汇算清缴后做一次案例库版本升级（v1.x → v2.x）
3. 重大政策变动（如新出台总局公告）后 30 天内同步更新影响到的案例
