## 一、项目背景

1. **政策环境**：财税〔2015〕119 号、国家税务总局公告 2017 年第 40 号、国家税务总局公告 2023 年第 7 号等系列文件已构建起较为完整的研发费用加计扣除政策体系，软件 / 互联网 / IT 行业一般企业适用 **100% 加计扣除**。
2. **公司现状**：公司年度研发投入超 1 亿元，对应每年潜在加计扣除节税金额约 **2500 万元**（按 25% 所得税率测算）。但当前存在三类突出风险：
   - **研发活动认定边界不清**：版本迭代 / 客户定制 / 运维支持等活动与"真研发"的边界长期由各方各自判断，业务部门与财务部门长期扯皮。
   - **人员工时分摊证据链薄弱**：研发人员普遍兼岗（售前 / 实施 / 运维），但工时记录靠周报 + 主管签字，缺乏客观证据，稽查抗辩力弱。
   - **组织职责不清**：加计扣除目前由财务部门一人扛，研发 / HR / 法务 / 内审等业务方未深度参与，体系无法落地。
3. **外部压力**：公司同时面临税务稽查（历史年度追溯）、年报审计、IPO / 持续披露三重合规要求，风险敞口持续扩大，亟需系统化、可审计、可演进的管理体系。

## 二、项目目标

### 2.1 总体目标

24 个月内建成符合"治理 + 轻量数据中台"双轮驱动架构的研发费用加计扣除风险管理体系，实现：
- **合规目标**：经得起税务稽查 + 年报审计 + IPO 披露三重检验
- **效率目标**：月度归集 5 个工作日内闭环；年度汇算清缴前 60 天完成自查
- **风险目标**：历史年度无重大追溯调整；未来年度仪表盘红灯指标 <10%

### 2.2 Phase 1（M0–M6，本章程重点）阶段目标

| # | 阶段目标 | 验收口径 |
|---|----------|----------|
| 1 | 治理层落地 | 委员会成立、加计办挂牌、RACI 矩阵全员认知 |
| 2 | 判定层落地 | 管理办法 + 判定矩阵 + 三道闸门正式发布 |
| 3 | 证据层 MVP 上线 | 飞书多维表格四张主表运行、Jira/Git/HR 接口稳定 |
| 4 | 工时填报全员上线 | 研发部 100% 入系统、过渡期满后剔除率 <20% |
| 5 | 内控试运行 | 连续 3 个月月度对账 5 项必对全部通过 |

## 三、项目范围

### 3.1 范围内（In Scope）

- 公司本部全部研发活动 / 研发人员 / 研发项目
- 治理 / 判定 / 证据 / 内控 / 演进 / KPI / 预案 共 7 层体系建设
- 飞书多维表格 + 应用引擎平台部署
- HR / Jira / Git 三类源系统的轻量接入
- 历史项目（M0 时点在行项目）补录与规范化

### 3.2 范围外（Out of Scope）

- 子公司 / 关联公司研发活动（待 Phase 3 评估后决定是否纳入）
- 重型 IT 系统集成（ERP 深度改造）——保留至 24 个月评估后再决策
- 高新技术企业认定 / 软件企业认定的口径合并管理（Phase 2 启动）
- 委外 / 合作研发合同专项治理（前置至 M7 启动，不在 M0–M6 范围）
